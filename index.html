履程：面向内地与香港，把履历对齐真实岗位

地区就业市场痛点

中国内地

  • 环境高度竞争，雇主极看重垂直实习与对口经历，履历与 JD 之间常见「匹配断层」。
  • 职业规划近乎刚需，传统高价咨询却将多数普通学生挡在门外，信息与服务仍不对称。

香港

  • 国际金融与多元产业并存，简历需要经得住行业逻辑的审视，而非简单信息堆砌。
  • 筛选与匹配成本高，求职者难以及时看清岗位叙事与自身材料的差距。

向下可展开产品闭环、技术选型与社会效益；能力由方舟大模型 + RAG + 流式交付支撑，把对齐 JD、补经历与可执行规划收成一条你能跟完的链路。

产品闭环、技术选型与社会效益

把散落在各处的经历，收成模型能读懂、敢下判断的一份叙事——再交回你手里签字。

产品闭环 六步 · 录入 → 规划
  1. 录入 — 简历与多份 JD 结构化入库，解析绑定会话，后续分析、改写、追问共用同一份可对齐的上下文。
  2. 分析 — 简历 × 多岗位匹配推理；长报告用 SSE 渐进送达，少白屏、多「看得见在推」。
  3. 改写 — 按教育 / 实习 / 项目等模块分段生成；口述经历经 ASR 再进大模型整理与改写，多模态进同一条管线。
  4. 补信息 — 模型驱动追问与选项，缺口补全后写回会话，再喂回生成链路。
  5. 核对 — 原文与改写并排、差异高亮，确认后再采纳，闭环可审计。
  6. 规划 — 模型归纳投递时间线与行动建议，长内容支持屏内滚动读完，从材料落到「下一步干嘛」。
技术选型 模型 · RAG · 工程栈
  • 火山引擎方舟（豆包）大模型 — OpenAI 兼容 API（北京区域端点），统一客户端与模型版本切换。
  • 检索增强（RAG） — 岗位描述与简历结构化片段检索相关上下文，再交由大模型做分析、追问与改写；索引形态随迭代对齐。
  • 多场景 prompt 与链式编排 — 分析、改写、追问、规划、语音整理等分场景模板与调用链（可配合 LangChain 组织复杂逻辑）。
  • SSE 流式分析 — 长上下文与多 JD 下服务端推送分段呈现，避免一次等待大段 JSON 的断裂感。
  • ASR + LLM — 火山语音识别与大模型总结 / 追问衔接；前端可选用 WebSocket 流式转写，口述快速进入结构化流程。

工程底座：React、TypeScript、Vite、Tailwind；FastAPI；Supabase（PostgreSQL）；前端 Vercel。交付与扩展的底子，差异化仍在模型与数据链。

社会效益 普惠 · 匹配 · 赋权

简历与 JD 错位、招聘初筛成本高，叠加高价职业规划，常把普通求职者挡在机会之外。履程用 AI 把深度对齐岗位、挖掘既有优势与实习路径建议做成可反复使用的辅助，压低门槛，让缺少资源背景的人也能持续获得像样的支持。

人与岗匹配更准，盲投与误筛略减，社会层面的人力资源配置会更健康；用技术降低信息与服务的门槛，也是在放大普通人的可见度与选择权。

分析

匹配分析：RAG 上下文 + 大模型推理

JD 又长又绕，多岗位并排时人眼先投降。要的是一眼看清「差在哪」，不是一份打不开的说明书。

简历与 JD 的结构化片段先检索对齐，再交给方舟大模型归纳差距与补强方向；前端用 SSE 流式接片段，配上摘要与维度图表。长分析可以边生成边读完，你始终知道系统正在往哪走，而不是对着空白干等。

匹配解读（演示长文）

多份 JD 并排时，系统先把共性要求抽成「岗位需求摘要」，再按岗位拆开看差异。RAG 检索把简历里可对齐的片段喂给大模型，避免模型在空白上下文里臆测你的经历。

下列段落模拟 SSE 流式推送累积后的长报告：你可以在本屏继续向下滚动读完，右侧演示区会随外层滚动同步下移，便于展示「长分析一口气读完」的阅读动线。

建议在实习与项目模块补充可验证指标（转化率、曝光、成本、时长等），并与目标岗位关键词做显式对齐；教育模块保持时间线闭合，减少解析歧义。

若目标岗位强调「ownership」，可把独立负责的范围写清边界：需求来源、协作角色、你拍板的决策点、上线后的复盘数据。模型在改写阶段会反复引用这些锚点。

演示站不连接后端；真实产品中，同一会话内的录入、分析、改写、追问与规划共用结构化上下文，减少重复粘贴与口径漂移。

补信息

模型驱动追问：把缺口补回上下文

经历写不细，模型不敢编；硬编了，你更不敢投。要的是它问你答，把洞补上，而不是一次 prompt 赌命。

大模型点名缺口，生成追问与可点选项；回答与简历片段一并写回会话，再驱动改写。你可选用手动输入逐条写细，或智能选项快速点选——人机协同把上下文补满，再进入下一轮改写。

核对

可审计的采纳闭环

黑盒改写不敢签字：你不知道它「优化」掉了什么真实经历。并排看清楚，再点采纳。

结构化 diff 标出改动范围，后端要点与说明可与并排视图对照，每一处改动有据可查。采纳前你能自己对齐「改了什么、为什么改」，心里踏实再保存。

规划

投递时间线：模型生成的行动草案

材料改完了仍不知道下周干嘛——焦虑从「写不好」变成「动不了」。要的是能执行的月历，不是鸡汤清单。

方舟大模型结合简历与前置分析出规划草案，经 plan API 落库，与录入—分析—改写串成闭环。从材料到行动可复看、可跟进，投递节奏落在同一条链路上。

规划草案

投递时间线（示意）

以下时间线与建议项为静态排版，演示长内容随屏内滚动同步阅读。

第 1–2 周 固化目标岗位清单，按 JD 共性补齐简历关键词与量化指标；完成 2 次小范围 peer review。
第 3–5 周 集中投递与内推并行，记录投递漏斗；按反馈迭代改写模块，优先实习与项目叙事。
第 6–8 周 复盘笔试/面试缺口，用追问模块补全上下文，再生成下一轮改写与规划补丁。

真实产品中，规划由 plan API 与会话进度共同驱动；此处仅展示版式与阅读节奏，不发起网络请求。

将「下一步行动」拆成可勾选里程碑，有助于把焦虑从「写不好」转移到「这周能完成哪三件小事」；演示页继续向下滚动即可读完本条与下文。

建议把内推、官网投递、招聘会三条渠道分开统计转化率；当某一渠道显著偏低时，优先检查简历关键词是否与该渠道 JD 风格匹配。

对文科背景候选人，模型侧通常会强化「叙事—证据—可验证指标」三段式；本静态页用于宣传站离线展示，不代表某次真实生成结果。
